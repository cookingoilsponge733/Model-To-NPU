# Разбор overhead в SDXL UNet

Краткий, но прикладной обзор того, где сейчас теряется wall-clock время в phone-side SDXL runtime.

## Почему split UNet пока неизбежен

Split здесь не является случайным дизайнерским решением.

- полный FP16 UNet весит примерно **5 GB**;
- практический лимит HTP по аллокации — около **3.5 GB**;
- текущие split-артефакты помещаются по отдельности:
  - encoder ≈ **2.52 GB**
  - decoder ≈ **2.69 GB**

То есть главная задача сейчас — не «любой ценой убрать split», а **сделать split-path дешевле по wall-clock времени**.

## Что показал глубокий probe

Замеры сделаны на живом phone runtime через `qnn-net-run` + `qnn-profile-viewer`.

### Стоимость одного half-step

| Сценарий | Wall time |
| --- | ---: |
| encoder single | **8312.0 ms** |
| decoder single | **8141.1 ms** |

### Эффект `mmap` на том же half-step

| Сценарий | База | С `--use_mmap` | Выигрыш |
| --- | ---: | ---: | ---: |
| encoder | 8312.0 ms | **6115.0 ms** | **26.4%** |
| decoder | 8141.1 ms | **6011.4 ms** | **26.2%** |

### Эффект repeat в одном процессе

| Сценарий | Single-call wall | repeat×4 avg/inference | Выигрыш |
| --- | ---: | ---: | ---: |
| encoder | 8312.0 ms | **3527.5 ms** | **57.6%** |
| decoder | 8141.1 ms | **4379.7 ms** | **46.2%** |

### Главный вывод

Самая тяжёлая потеря времени сейчас — это **overhead жизненного цикла runtime**:

- запуск процесса;
- загрузка context/binary;
- init/deinit графа;
- лишний file I/O вокруг каждого вызова.

`mmap` — это реальный и дешёвый выигрыш, но **не полное решение**.
Результаты repeat×4 очень явно показывают, что следующий большой шаг — это persistent runner / daemon.

## Уточнение по CFG

Текущий runtime уже не платит самый наивный путь `4 subprocess на шаг`.

Сейчас оптимизированный CFG-шаг выглядит так:

- **1 batched encoder call** для `uncond + cond`
- **1 batched decoder call** для `uncond + cond`

Поэтому CFG выше `1.0` всё ещё примерно удваивает реальную denoising-работу, но уже не повторяет старый худший паттерн подпроцессов.

## Что вошло в `v0.1.3`

В релизе `0.1.3` применён первый дешёвый runtime-win везде, где это важно:

- `phone_generate.py` теперь по умолчанию включает **`SDXL_QNN_USE_MMAP=1`**;
- rooted sample runtime зеркалит то же поведение;
- APK-path запуска теперь явно экспортирует `SDXL_QNN_USE_MMAP=1`;
- в репозиторий добавлены нормальные debug-утилиты:
  - `SDXL/debug/sdxl_speed_probe.py`
  - `SDXL/debug/sdxl_unet_overhead_probe.py`

При желании дефолт можно отключить для проверки:

```bash
export SDXL_QNN_USE_MMAP=0
```

## Follow-up по reuse в `v0.2.3`

Более поздний reuse-тюнинг phone runtime поменял уже не только итоговую цифру, но и саму *форму* замедления на первых шагах.

- первые четыре guided-шага UNet на текущем быстром CFG-пути теперь идут примерно как **12.2 → 10.4 → 9.9 → 9.8 s**;
- первые четыре шага в `CFG=1.0` no-guidance пути держатся примерно около **7.4 → 7.4 → 6.2 → 6.5 s** с обычной погрешностью между прогонами;
- практический README-visible APK marker держится на **78.0 s total** (Live Preview ON);
- более поздний точный runtime-прогон на том же `v0.2.3` пути (Live Preview OFF, обновлённый `phone_generate.py`) дал **62.0 s total** при `CLIP 1.787 s`, `UNet 55.980 s`, `VAE 3.138 s`.

То есть runtime уже не застревает в прежнем ощущении «почти плоской ~12-секундной полки» на первых guided-шагах. Теперь даже до дешёвого tail-пути видно реальное убывание warm-path стоимости.

## Практическая рекомендация

На короткой дистанции:

- держать `mmap` включённым по умолчанию;
- использовать новые probe-скрипты при любом изменении runtime;
- считать первый guided-шаг следующим главным горячим местом для оптимизации;
- внимательнее смотреть на overhead CLIP/prompt-path теперь, когда UNet и TAESD стали дешевле;
- не возвращаться к `8W8A`, пока не дожат runtime-overhead и lifecycle overhead.

На средней дистанции:

- делать persistent encoder/decoder runner, который загружает context один раз и переиспользует его между шагами.

## Следующие вероятные speed-win'ы после `v0.2.3`

После текущего reuse-прохода и переноса TAESD на GPU следующий набор практических целей выглядит так:

1. **parallel CLIP как low-risk эксперимент** — cond/uncond text-path теперь стал более заметной долей общего wall-clock времени, так что перекрытие части prompt-side работы уже стоит проверить;
2. **возврат к daemon / persistent-runner пути как к самому дорогому по выигрышу проекту** — repeat-in-one-process probe по-прежнему показывает, что lifecycle overhead остаётся главным налогом по wall-clock времени, даже несмотря на регрессию первой daemon-попытки;
3. **небольшие эксперименты с VAE/context warmup** — это вряд ли даст гигантский выигрыш, но после больших вопросов по UNet/CLIP их тоже стоит проверить.

Текущая практическая рекомендация такая: сначала дожимать первый guided-шаг UNet и CLIP overhead, но при этом не забывать, что daemon path всё ещё даёт самый большой теоретический upside, если удастся понять причину регрессии.
